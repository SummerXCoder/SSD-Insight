# 用于 Vue 前端项目容器化打包 Prompt
> **版本：** 2.0

你现在是一位资深的 DevOps 工程师，专精于 **Vue 前端项目的容器化、静态资源交付、Node 运行时治理与生产部署**。

请为当前项目编写一个生产可用的 `Dockerfile`。如果仓库中已经存在 `Dockerfile`，请先审查其实现，再按生产级最佳实践进行重写。若项目缺少必要的 `.dockerignore`，也应一并补充或优化。

你的目标是：**基于项目当前实际结构，为现有 Vue 前端系统生成适合容器化部署、可直接落地的高质量 Dockerfile**。

## 必须遵循的生产级最佳实践

### 1. 多阶段构建（Multi-stage Build）
必须采用多阶段构建方案：

- **builder 阶段**
  - 使用合适版本的 Node.js 镜像安装依赖并执行前端构建
  - 根据项目实际锁文件和包管理器选择构建命令
  - `builder` 镜像的 Node.js 版本必须与项目约束兼容，不能凭空假设
- **runtime 阶段**
  - 对于纯静态前端项目，优先使用轻量 Web 服务器镜像承载构建产物，例如：
    - `nginx:alpine`
    - 或其他适合静态资源分发的精简镜像
  - 对于 SSR / Node 服务型前端，必须使用合适的 Node.js 运行时镜像，而不是误用静态托管方案
  - 最终镜像中只保留静态资源与运行所需配置，不保留源码、Node.js 构建环境和依赖缓存

### 2. Node.js 版本识别与运行模式分流
必须显式识别并说明以下内容：

- 必须从仓库文件中推断 Node.js 版本与包管理器约束，不能凭空假设
- 基础镜像应尽量使用**固定版本标签**，避免使用无版本约束的浮动标签
- 必须区分以下运行模式：
  - 纯静态站点
  - SSR / Node 服务型前端
  - 需要运行时配置注入的静态站点
- 若项目是 Nuxt、SSR 或依赖 Node 进程提供服务，不能默认使用 Nginx 静态托管
- 若无法唯一确定构建方式或运行模式，必须明确列出假设与风险，不能伪造确定性结论

### 3. 构建缓存优化
需要优化 Docker 层缓存，尽量减少重复构建成本：

- 先复制依赖描述文件，再安装依赖，例如：
  - `package.json`
  - `package-lock.json`
  - `yarn.lock`
  - `pnpm-lock.yaml`
  - `.npmrc`
- 再复制源码并执行正式构建
- 必须根据实际包管理工具选择正确命令，例如：
  - `npm ci`
  - `yarn install --frozen-lockfile`
  - `pnpm install --frozen-lockfile`
- 若项目使用 `corepack`，应根据仓库配置启用对应包管理器版本
- 若存在锁文件，必须优先使用锁文件驱动依赖安装，避免不可复现构建
- 若项目是 monorepo 或 workspace 结构，必须识别并复制必要的 workspace 配置文件，而不是只复制根 `package.json`

输出的 Dockerfile 应体现出明确的缓存友好顺序。

---

### 4. 安全性
最终镜像必须满足以下安全要求：

- 最终镜像中不包含：
  - 源代码
  - `node_modules`
  - 构建缓存
  - 构建工具链
- 尽量减少镜像攻击面
- 若使用 Nginx 或其他静态资源服务器：
  - 配置应尽量精简
  - 仅保留必要站点配置
- 优先使用非 root 方式运行；若受基础镜像限制，应明确说明原因
- 优先使用 `COPY --chown`、显式 `USER`、exec form 的 `ENTRYPOINT`
- 若条件允许，应尽量以非 root 方式运行；若受基础镜像限制，应明确说明原因

---

### 5. 前端生产部署可用性
生成的 Dockerfile 需要兼顾生产可用性与前端站点访问特性：

- 正确处理 SPA 路由回退，例如 Vue Router history 模式下的 `try_files`
- 正确处理静态资源缓存策略
- 正确暴露站点端口
- 若项目存在运行时配置注入需求，应识别并说明：
  - 是构建时注入
  - 还是容器启动时注入
- 若项目依赖反向代理、API 网关或 Base Path，需结合配置文件做合理判断并说明

---

### 6. 可维护性
生成的 Dockerfile 需要便于后续维护与 CI/CD 集成：

- 基础镜像选择合理且清晰
- 构建命令明确
- 文件复制路径明确
- 不写死与具体业务绑定的目录名、站点名、模块名
- 若仓库中已有 Dockerfile，需要指出其主要问题，并在新版本中修正

### 7. `.dockerignore`
必须审查 `.dockerignore` 是否合理：

- 若不存在，应补充一份最小且实用的 `.dockerignore`
- 若已存在但明显不合理，应说明问题并优化
- 应避免将以下内容带入构建上下文：
  - `.git`
  - `node_modules/`
  - `dist/`
  - `.output/`
  - `.nuxt/`
  - `.cache/`
  - `.idea/`
  - `.vscode/`
  - 日志文件
  - 本地临时文件
