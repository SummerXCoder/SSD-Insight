## 1. 背景与目标

为企业提供一套会议室预约系统，解决会议室资源争用问题，确保预约不冲突，并通过审批流程管控资源使用。

**核心业务目标：**

- 用户可便捷预约会议室，系统保证同一会议室同一时间段不被重复占用
- 通过审批流程管控会议室使用
- 提供会议室信息管理、用户管理和使用统计能力

| 术语 | 英文 | 定义 |
|------|------|------|

| 会议室 | Meeting Room | 可被预约的共享空间资源，含容量、设备、位置等属性 |
| 用户 | User | 系统使用者，分为普通用户和管理员 |
| 普通用户 | Regular User | 可发起、修改、取消自己预约的用户 |
| 管理员 | Admin | 负责审批预约、管理会议室、管理用户、查看统计的用户 |
| 审批人 | Approver | 按部门指定的、负责审批该部门用户预约的管理员 |
| 部门 | Department | 扁平单层组织单元，关联审批人 |
| 时间段 | Time Slot | 预约的起止时间范围 |
| 冲突 | Conflict | 两个预约在同一会议室的时间范围上存在重叠 |
| 设备 | Equipment | 会议室配备的设施（如投影仪、白板、视频会议系统等） |
| 容量 | Capacity | 会议室可容纳的人数 |
| 停用 | Disabled | 会议室因维护等原因暂时不可预约的状态 |
| 站内信 | In-App Notification | 系统内部的消息通知机制，用户登录后可查看 |

| 预约禁令 | Reservation Ban | 因累计未到场次数达到阈值，对用户施加的禁止预约处罚 |

## 3. 用户与角色

| 角色 | 描述 | 核心权限 |
|------|------|----------|
| 普通用户 | 系统主要使用者 | 发起预约、修改自己的预约、取消自己的预约、签到、查看会议室列表与可用时间、查看站内信通知 |
| 管理员 | 系统管理者 | 审批预约（作为部门审批人）、增删改会议室、停用/启用会议室、管理用户、查看统计、查看站内信通知 |

**说明：** 管理员同时作为部门审批人，每个部门指定至少一名审批人。

## 4. 范围边界

### In Scope

- 用户注册/登录（账号密码）
- 会议室预约（含冲突检测）
- 预约审批流程（按部门指定审批人）
- 预约修改与取消
- 会议室信息管理（含容量、设备属性）
- 会议室筛选（按容量、设备等）
- 会议室停用/启用（停用时自动取消已有预约并通知）
- 站内信通知机制（审批结果、取消、停用等关键事件）
- 用户管理
- 使用统计（按时间、会议室、会议名称、人数、时长）
- 基本时间约束（提前≥1小时、时长≤2小时、可预约未来14天内）
- 会议签到（预约人手动签到，会议开始前后15分钟内）
- 未到场惩罚机制（阶梯式：累计3次未到场禁预约7天，累计5次未到场禁预约30天）

### Out of Scope

- 周期性/重复预约
- 会议室容量强制校验（预约时不强制参会人数≤容量）
- 多层部门层级与审批人继承
- 企业账号集成（LDAP/SSO等）
- 会议室预约费用/计费
- 会议室地图/导航
- 移动端专属适配
- 与日历系统（Outlook/Google Calendar等）集成
- 邮件/短信通知

## 5. 业务不变量 (Business Invariants)

| 编号 | 不变量 | 说明 |
|------|--------|------|
| INV-01 | 同一会议室同一时间段不可存在两条"已通过"或"待审批"的预约 | 冲突检测的核心约束；待审批预约也占用时间段，从根本上避免审批后冲突 |
| INV-02 | 预约必须经过审批通过方可生效 | 审批流程的强制约束 |
| INV-03 | 修改预约必须重新审批 | 修改规则的强制约束 |
| INV-04 | 停用的会议室不可被新预约 | 会议室可用性约束 |
| INV-05 | 预约时间必须满足基本约束（提前≥1小时、时长≤2小时、范围≤14天） | 时间约束的强制规则 |
| INV-06 | 会议结束后未签到的预约自动判定为"未到场" | 未到场判定的强制规则 |
| INV-07 | 处于预约禁令期间的用户不可发起新预约 | 惩罚机制的强制约束 |

## 6. 核心场景与功能需求

### SCN-01：用户预约会议室

- **目标：** 用户选择会议室和时间段，提交预约申请
- **触发条件：** 用户发起预约操作
- **关键流程：**
  1. 用户筛选/选择会议室
  2. 选择时间段
  3. 系统检测冲突（与已通过+待审批的预约比较）
  4. 若无冲突，预约状态设为"待审批"，该时间段被占用
  5. 系统通过站内信通知该部门审批人
  6. 审批人通过 → 预约状态变为"已通过"，通过站内信通知用户
  7. 审批人拒绝 → 预约状态变为"已拒绝"，释放时间段，通过站内信通知用户
- **业务规则：**
  - 提交时检测与"已通过"和"待审批"预约的冲突
  - 待审批预约占用时间段，其他人不可预约同一时间段
  - 时间约束：提前≥1小时、时长≤2小时、可预约未来14天内
- **异常场景：**
  - 提交时存在冲突 → 提示用户冲突信息，不允许提交
  - 时间约束不满足 → 提示具体约束违规

### SCN-02：用户取消预约

- **目标：** 用户取消自己的预约
- **触发条件：** 用户对已有预约执行取消操作
- **关键流程：**
  1. 用户选择自己的预约
  2. 执行取消
  3. 预约状态变为"已取消"，释放时间段
  4. 通过站内信通知审批人
- **业务规则：** 仅可取消自己的预约
- **异常场景：** 预约已取消 → 提示不可重复取消

### SCN-03：用户修改预约

- **目标：** 用户修改自己的预约信息
- **触发条件：** 用户对已有预约执行修改操作
- **关键流程：**
  1. 用户修改预约信息（时间/会议室/其他）
  2. 系统检测冲突（与已通过+待审批的预约比较，排除自身）
  3. 若无冲突，原时间段释放，预约状态重置为"待审批"，新时间段被占用
  4. 重新走审批流程
- **业务规则：** 任何修改都需重新审批
- **异常场景：** 修改后存在冲突 → 提示冲突信息，不允许提交修改

### SCN-04：预约被拒绝后重新提交

- **目标：** 用户基于被拒绝的预约修改后重新提交
- **触发条件：** 用户对"已拒绝"状态的预约执行修改操作
- **关键流程：** 同 SCN-03
- **业务规则：** 可修改后重新提交，无需从零填写

### SCN-05：管理员审批预约

- **目标：** 审批人审核本部门用户的预约申请
- **触发条件：** 审批人查看待审批列表
- **关键流程：**
  1. 审批人查看待审批列表
  2. 选择通过或拒绝
  3. 通过 → 预约状态变为"已通过"，通过站内信通知用户
  4. 拒绝 → 预约状态变为"已拒绝"，释放时间段，通过站内信通知用户
- **业务规则：** 待审批预约已占用时间段，审批通过无需复检冲突
- **异常场景：** 无（因待审批已占用时间段，不存在审批后冲突的情况）

### SCN-06：管理员管理会议室

- **目标：** 管理员维护会议室信息
- **触发条件：** 管理员执行会议室管理操作
- **关键流程：** 增删改会议室信息（名称、位置、容量、设备等）
- **业务规则：** 会议室信息变更不影响已有预约
- **异常场景：** 删除仍有未来预约的会议室 → 提示存在关联预约，需先处理

### SCN-07：管理员停用/启用会议室

- **目标：** 管理员将会议室设为停用或重新启用
- **触发条件：** 会议室需维护或维护完成
- **关键流程：**
  1. 管理员停用会议室
  2. 会议室状态变为"停用"，不可新预约
  3. 该会议室所有未来预约自动取消，释放时间段
  4. 通过站内信通知所有受影响的预约用户
  5. 启用时恢复可预约状态
- **业务规则：** 停用后不可新预约，已有预约自动取消
- **异常场景：** 无

### SCN-08：用户注册与登录

- **目标：** 用户通过账号密码注册和登录系统
- **触发条件：** 用户访问系统
- **关键流程：**
  1. 新用户注册（填写用户名、密码、所属部门）
  2. 已有用户登录（用户名+密码）
  3. 获取身份与权限
- **业务规则：** 用户注册时需指定所属部门

### SCN-09：管理员管理用户

- **目标：** 管理员管理系统用户
- **触发条件：** 管理员执行用户管理操作
- **关键流程：** 查看/编辑用户信息、分配角色、指定部门
- **业务规则：** 管理员可修改用户的角色和部门归属

### SCN-10：查看统计

- **目标：** 管理员查看会议室使用统计
- **触发条件：** 管理员访问统计页面
- **关键流程：** 查看按时间、会议室、会议名称、人数、时长等维度的统计数据
- **业务规则：** 仅管理员可查看统计

### SCN-11：用户签到

- **目标：** 预约人在会议开始前后签到，确认到场
- **触发条件：** 预约人在签到时间窗口内执行签到操作
- **关键流程：**
  1. 预约人查看自己的"已通过"且当前处于签到窗口内的预约
  2. 点击签到
  3. 系统记录签到时间，预约标记为"已签到"
- **业务规则：**
  - 签到时间窗口：会议开始前15分钟至会议开始后15分钟
  - 仅预约人本人可签到
  - 仅"已通过"状态的预约可签到
- **异常场景：**
  - 不在签到时间窗口内 → 提示签到未开放或已过期
  - 已签到 → 提示不可重复签到

### SCN-12：未到场自动判定与预约禁令

- **目标：** 对多次预约但未到场的用户施加预约禁令，防止资源浪费
- **触发条件：** 会议结束时间到达，预约人未完成签到
- **关键流程：**
  1. 会议结束后，系统自动检查该预约是否已签到
  2. 若未签到，预约标记为"未到场"，用户未到场次数+1
  3. 系统检查用户累计未到场次数：
     - 累计达到3次 → 对用户施加预约禁令，禁止预约7天
     - 累计达到5次 → 对用户施加预约禁令，禁止预约30天
  4. 通过站内信通知用户未到场判定结果及禁令状态
- **业务规则：**
  - 未到场次数为累计计数，不因禁令到期而清零
  - 禁令期间用户不可发起新预约（提交时检测并拒绝）
  - 禁令到期后自动解除，用户可正常预约
  - 取消的预约不计入未到场统计
- **异常场景：**
  - 用户处于禁令期间尝试预约 → 系统拒绝并提示禁令剩余天数

## 7. 验收标准 (Acceptance Criteria)

| 编号 | 验收标准 | 关联 |
|------|----------|------|
| AC-01 | **Given** 会议室A在10:00-11:00无已通过或待审批的预约 **When** 用户提交会议室A 10:00-11:00的预约 **Then** 预约状态为"待审批"，该时间段被占用，审批人收到站内信通知 | SCN-01, REQ-01 |
| AC-02 | **Given** 会议室A在10:00-11:00已有已通过或待审批的预约 **When** 用户提交会议室A 10:00-11:00的预约 **Then** 系统拒绝提交并提示冲突 | SCN-01, INV-01 |
| AC-03 | **Given** 预约处于"待审批"状态 **When** 审批人通过审批 **Then** 预约状态变为"已通过"，用户收到站内信通知 | SCN-05, INV-02 |
| AC-04 | **Given** 预约处于"待审批"状态 **When** 审批人拒绝 **Then** 预约状态变为"已拒绝"，时间段释放，用户收到站内信通知 | SCN-05 |
| AC-05 | **Given** 用户有已通过或待审批的预约 **When** 用户取消该预约 **Then** 预约状态变为"已取消"，时间段释放，审批人收到站内信通知 | SCN-02 |
| AC-06 | **Given** 用户有已通过的预约 **When** 用户修改预约信息且新时间段无冲突 **Then** 原时间段释放，预约状态重置为"待审批"，新时间段被占用，需重新审批 | SCN-03, INV-03 |
| AC-07 | **Given** 用户修改预约信息 **When** 新时间段与已有预约冲突 **Then** 系统拒绝修改并提示冲突 | SCN-03, INV-01 |
| AC-08 | **Given** 预约被拒绝 **When** 用户修改后重新提交且无冲突 **Then** 预约状态变为"待审批"，走审批流程 | SCN-04 |
| AC-09 | **Given** 用户提交预约时间不满足约束 **When** 提前量<1小时或时长>2小时或超出14天范围 **Then** 系统拒绝提交并提示具体约束违规 | SCN-01, INV-05 |
| AC-10 | **Given** 会议室处于"停用"状态 **When** 用户尝试预约该会议室 **Then** 系统拒绝预约 | SCN-07, INV-04 |
| AC-11 | **Given** 管理员停用会议室 **When** 该会议室已有未来预约 **Then** 所有未来预约自动取消并释放时间段，相关用户收到站内信通知 | SCN-07 |
| AC-12 | **Given** 用户未登录 **When** 尝试访问预约功能 **Then** 系统要求登录 | SCN-08 |
| AC-13 | **Given** 管理员启用已停用的会议室 **When** 启用操作完成 **Then** 会议室恢复可预约状态 | SCN-07 |
| AC-14 | **Given** 预约已通过且当前时间在会议开始前15分钟至开始后15分钟内 **When** 预约人执行签到 **Then** 预约标记为"已签到" | SCN-11 |
| AC-15 | **Given** 当前时间不在签到窗口内（会议开始前>15分钟或开始后>15分钟） **When** 预约人尝试签到 **Then** 系统提示签到未开放或已过期 | SCN-11 |
| AC-16 | **Given** 会议已结束且预约人未签到 **When** 系统自动判定 **Then** 预约标记为"未到场"，用户累计未到场次数+1 | SCN-12, INV-06 |
| AC-17 | **Given** 用户累计未到场次数达到3次 **When** 系统检查 **Then** 对用户施加预约禁令7天，用户收到站内信通知 | SCN-12, INV-07 |
| AC-18 | **Given** 用户处于预约禁令期间 **When** 用户尝试提交新预约 **Then** 系统拒绝并提示禁令剩余天数 | SCN-12, INV-07 |

## 10. 风险、假设与待确认项

| 编号 | 类型 | 内容 | 影响范围 |
|------|------|------|----------|
| RSK-01 | 风险 | 待审批预约占用时间段，若预约最终被拒绝，该时间段被无效占用，可能降低会议室利用率 | SCN-01, SCN-05 |
| ASM-01 | 假设 | 预约状态流转：待审批→已通过/已拒绝；修改→待审批；取消→已取消；已通过→已签到/未到场 | 全局 |
| ASM-02 | 假设 | 预约需填写会议名称和参会人数（用于统计），但不强制校验人数≤容量 | SCN-01 |
| ASM-03 | 假设 | 未到场次数为终身累计，不因禁令到期而清零；管理员可手动清零 | SCN-12 |
| RSK-02 | 风险 | 手动签到无法防止代签，可能存在虚假签到 | SCN-11 |
