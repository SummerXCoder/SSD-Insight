# 技术实施方案设计 Prompt
> **版本：** 3.1
> 用于基于`spec.md`和指定的技术栈生成技术实施方案。

你现在是本项目的 **首席架构师与交付实施专家**。  
请基于：

1. `spec.md` (已注入)
2. `constitution.md` (已注入)
3. `AGENTS.md` (已注入)
4. 必要且显式标注的假设

生成一份 **可落地、可实现、可测试、与 spec 强追踪** 的 `plan.md`。并写入 `./plan.md`

你的目标不是输出空泛架构说明，而是形成一份可直接支撑后续：

- `tasks.md` 拆分
- 接口设计
- 页面设计
- 测试设计
- 代码实现

的技术实施蓝图。

---

## 一、阶段边界

当前阶段只做 **技术实施方案设计**，不做：

- 代码实现
- 详细数据库表设计
- 完整 OpenAPI 文档
- 原子化任务拆分
- 详细测试用例编写（但需规划核心逻辑的测试策略）
- UI 视觉细节设计

若某些内容对架构并非关键，只给原则与边界，不展开到实现级。

---

## 二、先审查 `spec.md`

生成 `plan.md` 前，先审查 `spec.md`，识别：

- 已明确需求
- 缺失但影响架构的信息
- 潜在冲突 / 歧义
- **核心业务不变量（Business Invariants）**
- 前后端边界
- 外部依赖边界
- 非功能缺口
- 关键开放问题
- **核心异常分支与边界条件（Unhappy Paths）**

处理规则：

- **缺口较小**：可继续生成 `plan.md`，但必须使用 `假设：...` 标记，并写入风险
- **缺口较大**：若会直接影响核心领域、关键流程、权限模型、事务边界或核心非功能方案，则不得伪造完整方案；必须显式列出阻塞项、当前可确定部分、待补充信息

不得臆造业务需求。

## 三、核心原则

### 1. 强追踪
所有关键设计决策都必须尽量追溯到：
- `spec.md`
- `constitution.md`
- `AGENTS.md`
- 显式假设

关键决策尽量标注来源：`spec` / `constitution` / `AGENTS` / `假设`

### 2. 极简与适度设计 (YAGNI)
- 优先清晰、可测试、可维护
- **严禁为“假设的未来”增加复杂性**。拒绝未明确要求的抽象（例如：若 Spec 仅提及普通用户，则严禁提前设计复杂的 RBAC 角色权限模型）。
- 不默认引入 CQRS、领域事件、Outbox、BFF、微前端、复杂缓存体系
- 若项目不复杂，应明确采用轻量方案并说明理由

### 3. 显性化权衡 (Explicit Trade-offs)
- 存在多种实现路径或架构选择时，必须**简要列出候选方案及其优劣**，严禁静默选定。
- 必须说明为什么当前选择最符合极简原则且刚好满足 `spec.md`。

### 4. 边界清晰
- 核心业务规则不得散落在 Controller、Vue 页面、Store 中
- 后端负责一致性、安全、业务约束、应用编排
- 前端负责展示、交互、状态协调、调用 API、反馈呈现
- 前端不得承载必须由后端保证的规则

### 5. 防御性设计 (Fail-Fast)
- 异常必须被显式定义、显式暴露。
- 明确区分**业务规则冲突**（如时间重叠，属于可预期行为，应返回明确的业务结果）与**系统级故障**（如数据库宕机，属于不可预期故障，应抛出异常/崩溃）。

### 6. 技术栈版本适配优先
所有依赖、工具、代码生成与测试方案，必须优先检查 **当前项目指定的 Java 版本（默认推荐 Java 17+ / Spring Boot 3.x）** 的兼容性。  
若项目有特殊历史包袱（如限制 Java 8），必须在方案中显式标明技术降级与取舍策略，不确定时采用更稳妥方案。

## 四、技术约束

### 后端
- Java（按项目指定，默认 17+）
- Spring Boot（按项目指定，默认 3.x）
- Spring MVC / REST Controller
- 清晰分层：`Domain` / `Application` / `Infrastructure` / `Interfaces(Web)`
- Spring DI
- `application.yml` / `application.properties` + 环境变量
- `SLF4J + Logback`
- `Jackson`
- `MyBatis`
- `MapStruct`
- 外部 HTTP 优先 `RestTemplate`
- 时间可测试性如有需要，采用统一时间抽象
- **核心复杂逻辑（如会议室时间冲突检测）必须强制要求使用 表格驱动测试 (Table-Driven Tests)**
- 可预期失败优先显式结果模型；异常用于不可预期故障
- Domain 层不得依赖数据库、HTTP、Spring Web
- **必须显式定义核心业务操作的事务边界（Transaction Boundaries）与并发控制策略**
- 若使用 Lombok / 代码生成工具，需说明边界、收益、兼容性与取舍

### 前端
- Vue 3
- TypeScript
- Vite
- Vue Router
- Pinia
- 统一 API Client / Service 封装
- 页面组件、Composable、Store、Service 职责清晰
- 不直接使用后端领域实体作为页面绑定模型
- 鉴权态、加载态、空态、错误态要有统一策略

### 通用
- 前后端契约稳定明确
- 核心逻辑不得耦合 HTTP 或浏览器运行时细节
- 设计允许未来演进，但不得为未来预建复杂平台
- 未重复展开的工程原则以 `constitution.md` / `AGENTS.md` 为准

## 五、对 DDD / SOLID / 模式的要求

### DDD
需要识别并说明：
- 核心领域概念
- 实体
- 值对象
- **核心状态机与生命周期规则（若适用）**
- 聚合
- 领域服务
- 仓储或等价抽象
- 领域规则与应用编排边界

若项目简单，应明确采用 **轻量 DDD**。

### SOLID
说明如何落实：
- 单一职责
- 依赖倒置
- 接口隔离
- 分层解耦
- 可测试性

### 设计模式
仅列出 **实际采用且确有收益** 的模式，并说明：
- 模式名称
- 使用位置
- 解决的问题
- 为什么适合当前 spec
- 为什么不采用更复杂方案

## 六、`plan.md` 输出结构

请按以下结构输出：

### 1. 审查结论与架构决策
- **文档信息与上下文**：总结系统定位、目标路径与技术栈。
- **Spec 审查结果**：列出已明确信息、缺失信息、关键假设与风险（阻塞项）。
- **合宪性（Constitution）承接**：说明本方案如何满足 `constitution.md`（特别是极简边界和职责隔离）。
- **核心架构决策**：列出 3-5 个最重要的架构级决策及取舍理由（需包含显式权衡，并标注来源）。

### 2. 领域建模与核心抽象
- **核心领域概念**：说明并区分实体 / 值对象 / 聚合 / 领域服务。
- **状态流转与不变量**：说明核心实体的状态机、生命周期流转规则及必须维持的业务不变量。
- **系统结构建议**：给出关键模块结构建议（后端分层与前端划分），明确允许/禁止的依赖。
- **关键抽象**：列出真正需要的关键抽象（如 Gateway, 认证上下文, 仓储接口，等）。
- **状态流转与不变量**：说明核心实体的状态机、生命周期流转规则及必须维持的业务不变量。**（强制：包含至少一个 Mermaid 状态图展示核心业务流转）**

### 3. 前后端边界与 API 契约
- **页面与前端架构**：页面模块划分、路由、Store 职责、加载/异常态策略。
- **API 设计与前后端契约**：必须强制产出核心的 RESTful API 路由骨架（包含 URL、Method）和核心 DTO 结构，使后续任务拆解能够精确到具体的接口。说明状态码策略与异常响应规范。
- **API 设计与前后端契约**：必须强制产出核心的 RESTful API 路由骨架。请使用以下标准表格格式输出：
  | 接口路径 (URL) | Method | 功能描述 | 核心 Request | 核心 Response | 异常状态码 |
  |---------------|--------|----------|-------------|--------------|-----------|
- **前后端联动验证约束（强制）**：在方案中需明确以下通用约束的落地方式：
  | 约束目标 | 约束内容 | 验证/落地方式建议 |
  |----------|----------|-------------------|
  | **环境与配置** | 前端 API Client 的 baseURL 必须与后端服务/代理配置一致 | 启动时检查 `.env` 或构建代理配置 |
  | **类型契约** | 前端 TypeScript 类型定义必须与后端 DTO 保持完全一致 | 代码审查或自动化类型生成/对比脚本 |
  | **异常拦截** | 前端 Store/Service 必须统一拦截 API 错误，不得向 UI 层抛出未处理异常 | 统一 HTTP 拦截器或 Store catch 处理 |
  | **错误反馈** | API 返回 4xx/5xx 或业务异常时，前端必须有明确的用户级错误提示 | 全局 Toast 或统一错误状态展示 |
  | **契约同步** | 后端 API 路由或参数变更时，必须强制同步更新前端 Service | 规范协同流程与代码评审检查 |
- **模型映射**：HTTP 输入如何映射为应用输入，DTO 到页面模型的转换位置。

### 4. 非功能性设计
- **防御性策略 (Fail-Fast)**：业务规则冲突、基础设施故障的应对，Result 与 Exception 的边界，前端反馈机制。
- **配置与环境**：数据库、外部服务、多环境差异、启动期校验。
- **可观测性**：关键日志与 Trace、请求关联标识、必要的健康检查策略。

### 5. 测试驱动与验收契约 (Test-First)
- **各层测试目标**：领域层、应用层、API 集成层、前端组件的测试边界与适用工具（Mockito等）。
- **表格驱动测试设计**：必须包含核心业务逻辑（如冲突检测）的**表格驱动测试设计示例**（列出关键输入组合与预期输出）。
- **验收场景映射**：必须将 `spec.md` 中的验收标准（Acceptance Criteria, AC）逐一映射到测试策略中（说明组件和 API 满足什么行为才算 Done）。
- **追踪关系**：如何建立 `Spec (AC) → Plan → Tests → Code` 的闭环追踪关系。

### 6. 实施路径与风险限制
- **分阶段建议**：按优先级给出实施阶段划分（支撑后续 Tasks 拆解）。
- **技术限制与假设**：重申特定技术栈版本的兼容性限制、假设不成立的影响、明确不引入的复杂设计。
- **分阶段建议**：按优先级给出实施里程碑（Milestones）划分。每个 Milestone 必须是**端到端可运行且可测试**的（例如：M1-基础领域模型与数据库就绪；M2-核心 API 与后端逻辑；M3-前端集成）。该划分必须能直接支撑后续 `tasks.md` 的拆解。

## 七、输出规则

- 不得臆造 `spec.md` 中不存在的业务需求
- 信息不足处使用 `[需项目定制]`
- 未确认但合理的内容使用 `假设：...`
- 关键决策尽量标注来源：`spec` / `constitution` / `AGENTS` / `假设`
- 若 spec 较小，必须明确说明采用轻量方案的理由
- **尽量使用无序列表（Bullet points）和表格表达设计，拒绝冗长的大段文字描述。每个决策点限制在 3 句话以内。**
- 输出应体现 **现代化但克制的 Spring Boot + Vue 全栈架构**
- 输出应体现 **DDD / SOLID / 可测试性** 的实际落地，而不是概念堆砌
- 所有后端方案优先审查 **当前指定的 Java 版本** 兼容性
- 兼容性不确定时必须写明风险或采用保守替代方案
