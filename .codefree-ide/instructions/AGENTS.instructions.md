# AGENTS.md（项目执行协议 / SOP）

> **版本：** 3.2
> **地位：** AI Agent 的执行规范。
> 本文件承接 `constitution.md`，将宪法转化为可执行规则。

---

## 1. 编码前检查与裁决：不假设，不隐藏，多博弈

在执行任何任务前，Agent 必须先判断任务类型，并确认对应依据：
1. **正常开发任务**：必须确认当前 Task ID、业务规则 (`spec.md`) 与技术约束 (`plan.md`)。
2. **棕地发现任务**：若任务目标是补全文档、重建事实或澄清遗留系统行为，可暂以既有代码、运行行为、数据库结构、接口响应与运维事实作为输入，但产出必须首先是待评审的文档结论，而不是直接实现变更。
3. **词汇对齐**：识别并提取 `spec.md` 中的核心领域词汇；若 `spec.md` 缺失，则先在事实发现阶段提炼候选领域词汇，并在文档评审通过后再用于正式实现。
4. **验证方案**：成功标准是什么？验证方式是什么？正常开发任务必须直接对应 `spec.md` 的验收标准；棕地发现任务必须提供可复核的证据来源。
5. **澄清不确定性**：
   - **拒绝盲从**：若需求模糊或过度设计，必须推敲（Push back）并呈现更简单的替代方案。
   - **暴露歧义**：若存在多种合理解释，必须呈现选项而非静默选定。
   - **停止疑惑**：若存在影响实现或验收的关键不确定性，严禁直接开工，必须先澄清。澄清问题时，必须提供带上下文的选项（如：选项A符合规范但成本高；选项B是临时妥协但极简），交由人类做单选题，严禁开放式反问。

**双重裁决原则**：
- Agent 的行为必须同时符合 **治理约束** (`constitution.md` / `AGENTS.md`) 与 **事实约束** (`spec.md` / `plan.md` / `tasks.md`)。

## 2. 目标驱动执行：定义成功，闭环交付

复杂任务开始前，输出简要的**执行计划**（必须短小、可验证）：
> 1. [步骤 A] → 验证方式：[检查点]

每个任务必须形成**闭环**：
> 理解任务 → 确认依据 → 编写验证 → 最小实现 → 运行验证 → 文档同步检查 → 风险说明

**强制文档同步 (Sync) 规则：**
如果在编码过程中发现实际技术限制导致必须偏离 `plan.md` 或 `spec.md`，Agent 不得直接妥协实现，必须先触发熔断上报；若获人类授权修改，必须先更新对应权威文档，再恢复编码。

---

## 3. 极简主义与外科手术式修改：触碰最小化，产出最优化

**极简原则：**
- **需求锁定**：Agent 不得实现未被明确要求的功能。严禁为了“未来扩展性”增加配置、抽象或插件。
- **代码密度**：追求逻辑的高密度。如果 50 行能清晰解决的问题写了 200 行，必须重写。
- **边界克制**：不对理论上不可能发生的场景编写冗余的错误处理逻辑。
- **方案优先**：如果发现更简单且满足文档约束的路径，必须提出并优先采用。

**外科手术式修改：**
- **禁止顺手**：重构、格式化、改注释、升级依赖、修复无关历史问题。只触碰当前任务必须触碰的部分。
- **清理垃圾**：必须移除本次修改产生的临时代码，以及由于本次修改而导致不再被使用的 imports、变量、函数（清理“孤儿”）。**注意：清理孤儿代码前，必须通过全局搜索 (Grep) 或静态分析工具确认其在整个项目中无引用，避免误删。**
- **保留遗留**：除非被明确要求，否则不要清理代码中原有的“遗留垃圾”（Dead Code）。

---

## 4. 测试、验证与错误处理：先证后码，显式暴露

**测试与验证：**
- **新功能 / Bug 修复**：先写失败测试（行为/契约/复现测试），再实现，最后验证通过。**测试用例的描述和命名必须直接引用 `spec.md` 中的验收标准。**
- **重构**：必须有保护测试。重构前后测试均需通过，且不得改变外部行为。
- **非代码任务**：提供匹配的验证证据（如 CI dry-run、静态检查）。

**错误处理（Fail-Fast）：**
- **禁止沉默失败**。错误必须显式暴露、显式处理、保留上下文。
- 严禁吞异常、返回语义不明的 `null`、用默认值掩盖错误或在错误状态下继续执行。

## 5. 熔断机制

**必须熔断（停止实现并输出《熔断报告》）的情况：**
- 缺少依据：对于正常开发任务，`spec.md` 或 `plan.md` 缺失、冲突或不可验收；若当前任务属于棕地发现或文档重建，则必须先停止实现，转为事实发现与文档补全流程。
- 扩大范围：当前任务需要扩大范围（非编译/测试修复）才能完成。
- 改变事实：需要改变业务规则、外部行为、接口契约或关键技术方案。
- 重大风险：发现数据一致性、安全性或生产稳定性重大风险。

**《熔断报告》格式：**
- 熔断原因 / 涉及文档与代码 / 影响范围 / 建议修正路径 / 需要决策的问题

---

## 6. 一句话原则与规范指标

**原则：** 先以怀疑论者审视需求，再以外科医生精准修改，最后以法官姿态验证交付。

**规范生效指标：**
- Diff 中产生更少的不必要变更。
- 因为过度设计而导致的重写次数减少。
- 澄清性的问题出现在编码之前，而不是错误发生之后。
