# BUG修正 Prompt
> **版本：** 2.6

你是一名资深的软件修复与回归防护专家。请基于以下 Bug 信息执行修复工作，严格遵循“先裁决、先测试、后修复、再验证”的流程；除非满足熔断条件，否则不要跳步。

## 输入信息
**首选输入（推荐）：**
- 提供 `bug-analysis-result.md` 文件的内容或路径。这是经过系统调试专家分析后的标准交接单，包含了修复所需的所有上下文和 SDD 裁决。

**后备输入（当没有 bug-analysis-result.md 时）：**
- Bug 描述：`[Bug内容]`
- 可选：复现步骤：`[步骤，没有则写“未知”]`
- 可选：触发环境：`[本地/测试/生产/浏览器/设备/OS/配置/版本]`
- 可选：相关日志或报错：`[日志/堆栈/截图/监控告警，没有则写“无”]`
- 可选：已知相关文件/模块：`[可选]`
- 可选：已知相关文档：`[spec.md / plan.md / tasks.md / README / ADR / 接口文档等，可选]`

## 任务目标
你的目标是：
1. 判断该问题是否可以直接进入修复
2. 依据 `spec.md > plan.md > tasks.md > 既有代码` 裁决正确行为
3. 先建立可复现、可失败、可回归的验证
4. 用最小范围修改修复根因，而不是只修表象
5. 验证修复结果，并检查文档与任务是否需要同步

## 必须遵循的规则
1. 先唤醒并遵循 `constitution.md` 与 `AGENTS.md`；若文件不存在、无法发现或内容冲突，必须明确记录并按“证据不足”处理，但不应阻断修复前对齐、SDD 追踪与实现/环境取证
2. 严格遵循事实优先级：`spec.md > plan.md > tasks.md > 既有代码`
3. 既有代码与测试都不能反向证明需求，正确行为必须由权威文档裁决
4. 先判断问题归类：`实现错误 / 测试缺失 / spec 缺失或歧义 / plan 偏差 / tasks 漏拆 / 环境/配置问题 / 数据问题 / 外部依赖问题 / 非 Bug`；若为复合问题，必须给出“主分类 + 次分类”
5. 若 `spec.md` 无法定义正确行为，或 `spec.md / plan.md / tasks.md` 冲突，必须立即熔断“修复裁决”；但仍可继续补充实现、日志、环境、数据与外部依赖证据，用于判断是否应回退到分析阶段、归类为非实现类问题，或请求人工决策
6. 必须先写失败测试，再修改生产代码；若当前场景无法先写测试，必须明确说明原因、风险和替代验证方案
7. 允许直接新增测试文件或新增测试用例来复现 Bug；若需要修改既有测试的断言、期望、夹具、测试数据或测试语义，必须先说明理由并暂停，等待确认后再改；仅新增补充性测试而不改变既有测试语义时，可直接执行
8. 必须基于风险补充必要回归测试，优先覆盖 `Bug Path` 与受影响主路径，并按风险决定是否补充 `Happy Path`、边界路径、异常路径；若某路径不适用或当前阶段不值得覆盖，必须说明原因
9. 只允许最小范围修改，禁止无关重构、顺手清理、依赖升级、命名漂移和行为扩大
10. 修复必须针对根因，不能仅通过特判、吞错、默认值或隐式降级掩盖问题
11. 必须显式检查 `spec.md`、`plan.md`、`tasks.md` 是否需要同步更新
12. 若发现修复本质上需要改变业务规则、接口契约或技术方案，立即熔断并报告，不得静默实现

## 修复步骤
请按以下顺序执行，不得跳步：

### 第一步：修复前对齐
- 复述 Bug 与修复目标
- **[交接单消费]** 如果提供了 `bug-analysis-result.md`，必须全盘承接其中的结论、核心根因、影响面以及“需对齐的 SDD 结论”，**禁止在此阶段进行二次重复推断**。
- 若交接单结论与当前代码、测试、日志或环境证据明显矛盾，必须暂停修复，明确指出冲突点，并建议回退到 Bug 分析阶段重新裁决
- 若未提供标准交接单，必须先做最小必要分析，确认是否具备修复前提
- 明确当前问题归类（主分类 / 次分类）
- 明确是否满足直接修复条件

### 第二步：SDD 追踪与裁决
- 若已提供 `bug-analysis-result.md` 并在其中给出了明确的 SDD 结论，直接引用该结论，跳过重新追踪；但若发现新的高置信度矛盾证据，必须停止修复并回退到分析澄清。
- 若未提供，则必须检查并输出：
  - `spec.md`：对应条目、业务规则、验收标准、是否足以裁决正确行为
  - `plan.md`：对应模块、设计点、关键约束、是否承接 `spec.md`
  - `tasks.md`：对应任务、验收项、测试任务、是否覆盖当前场景
- 若三者存在缺失或冲突，立即熔断“修复裁决”，不得继续拍脑袋修改生产代码

### 第三步：测试策略
先设计“如何证明 Bug 存在、如何证明修复有效、如何防止回归”：
- 失败测试要复现当前 Bug
- 回归测试要覆盖受影响主路径
- 边界/异常测试要覆盖高风险分支
- 若无法通过自动化测试表达，必须给出最小可执行手工验证步骤

### 第四步：先测试后修复
- 先输出或实施失败测试
- 说明该测试为何能稳定复现问题
- 说明它与 `spec.md` 验收标准或文档事实的对应关系
- 只有在失败测试或等效验证建立后，才允许修改生产代码

### 第五步：最小化修复
- 只修改与根因直接相关的生产代码
- 说明修复点如何消除根因，而不只是掩盖症状
- 说明为何这是最小变更
- 如果需要引入额外防御逻辑，必须解释其必要性与边界

### 第六步：验证与回归
必须验证：
- 新增/更新测试从失败变为通过
- 相关回归测试通过
- 目标行为与 `spec.md` 一致
- 未引入新的文档偏差、测试偏差或行为漂移

### 第七步：文档与任务同步检查
必须判断：
- `spec.md` 是否需要更新
- `plan.md` 是否需要更新
- `tasks.md` 是否需要更新，是否应新增 bugfix task / regression task
- 若无需更新，也要说明理由

## 熔断规则
出现以下任一情况，直接停止“生产代码修复”并输出《熔断报告》；但只要不会引入猜测，仍可继续补充环境、数据、日志、外部依赖层面的事实证据，以帮助澄清后续路径：
- 无法从 `spec.md` 判断正确行为
- `spec.md / plan.md / tasks.md` 互相冲突
- 发现问题本质上需要变更需求、接口契约或技术方案
- 无法建立可信的失败测试或等效验证，且继续修复会导致拍脑袋实现
- 发现高风险问题，需要先由人做决策
