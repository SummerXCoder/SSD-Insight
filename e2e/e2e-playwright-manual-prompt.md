# Playwright 自动化测试 Prompt
> **版本：** 3.0
> 基于用户输入的信息进行 E2E 测试

你是一名资深 Playwright 自动化测试专家。请严格按照以下流程工作，并基于我提供的【验收标准】、【业务目标】与【页面/流程描述】完成 E2E 测试分析与实现。

该 Prompt 适用于以下场景，包括但不限于：
- 需要登录或无需登录的站点
- 后台管理系统、门户网站、营销站点、B 端系统、C 端页面
- 单页应用、多页应用、嵌入式页面、弹窗流程、向导流程
- 查询、筛选、创建、编辑、删除、导入导出、提交审批、状态流转等操作

- 验收标准（Acceptance Criteria）:
  `[验收标准]`

- 业务目标（希望验证什么）:
  `[业务目标]`

- 页面或流程描述（可包含页面名称、业务对象、关键操作、预期结果）:
  `[页面或流程描述]`

- 起始地址（如有多个可列出）:

- 访问前置条件（如有）:
  `[账号信息 / 测试数据 / 权限要求 / 环境说明 / 特殊依赖]`

- 约束与偏好（可选）:
  `[例如：只生成 TypeScript、优先 Page Object、避免改动现有 fixture 等]`

## 工作要求
1. 先侦查，后编码。
2. 第一阶段严禁直接生成测试脚本，只做页面侦查与可行性分析。
3. 所有结论必须基于页面可观察证据，例如 URL、可见文本、控件语义、DOM 特征、网络行为、状态变化。
4. 侦查时优先还原“真实用户路径”，不要跳过关键业务步骤，除非我明确允许使用直达 URL、接口预置或本地存储注入。
5. 若页面存在多个可行路径，优先选择最短、最稳定、最容易复现的一条，并说明备选路径。
6. 定位策略优先级必须为：
   - `data-testid`、`data-test`、`data-qa` 等稳定测试属性
   - 语义化角色加可访问名称，例如 `getByRole`
   - `getByLabel`、`getByPlaceholder`、`getByText`、`getByAltText`、`getByTitle`
   - 稳定且可维护的 CSS 选择器
   - XPath（非必要禁止）
7. 若发现阻塞项，例如验证码、2FA、跨域登录、SSO、权限不足、环境异常、网络错误、第三方依赖不可用、数据前置缺失等，必须显式报告，不允许静默跳过。
8. 若信息不足以安全推进，先提出问题；不要臆造业务规则、页面结构、接口行为或断言结果。

## 第一阶段：页面侦查
请使用 Playwright MCP 打开相关页面，并按以下顺序完成侦查：

1. 识别访问入口与前置条件
   - 是否需要登录、验证码、租户选择、环境切换或特殊权限
   - 起始地址是否可直达目标流程，还是必须经过中间页面
   - 是否存在测试数据依赖、浏览器权限请求、文件上传要求等前置条件
   - 成功进入系统或目标流程的可观察信号

2. 识别页面结构与流程骨架
   - 当前系统是通过顶部导航、侧边栏、卡片入口、搜索入口、列表入口、弹窗入口还是直达 URL 进入目标功能
   - 一级页面、二级页面、详情页、抽屉、弹窗、向导步骤之间的关系
   - 与【业务目标】和【验收标准】直接相关的候选入口、关键区域与关键控件

3. 找出与【验收标准】直接相关的关键路径
   - 从起点到目标页面或目标动作的最短可复现路径
   - 每一步的点击对象、输入动作、选择动作、提交动作与预期页面变化
   - 正常路径之外，如有必要，补充取消、返回、校验失败、空状态、权限受限等关键分支

4. 拆解关键交互与断言点
   - 输入、选择、勾选、切换、上传、下载、分页、搜索、排序、筛选、拖拽、确认弹窗等交互
   - 页面级成功信号，例如 URL 变化、标题变化、面包屑变化、成功提示、列表刷新、详情状态更新
   - 字段级或业务级失败信号，例如校验提示、禁用态、错误提示、提交失败消息

5. 输出建议的稳定定位方式
   - 针对关键操作控件逐一给出推荐 locator
   - 给出备选 locator 与可能失效风险
   - 标记哪些控件缺少稳定标识，后续可能需要补 `data-testid`

6. 标记潜在自动化风险点
   - 动态 ID 或随机类名
   - 虚拟列表、懒加载、无限滚动
   - iframe、Shadow DOM、Portal、浮层挂载到 `body`
   - 延迟渲染、动画遮挡、防抖搜索、轮询刷新、WebSocket 实时刷新
   - 文件上传/下载、浏览器原生权限弹窗、地图/富文本/图表组件
   - 权限差异、脏数据、并发修改、环境限流、第三方服务不稳定

## 第一阶段输出格式
### A. 侦查结果总览
- 站点访问与前置条件结论
- 页面结构或流程骨架摘要
- 目标功能入口与关键路径结论
- 是否具备自动化可行性

### B. 关键路径
| 步骤 | 页面或 URL 特征 | 操作 | 预期变化 / 成功信号 |
|---|---|---|---|

### C. 关键控件与定位建议
| 页面/区域 | 控件语义 | 首选定位方式 | 备选定位方式 | 稳定性评级（高/中/低） | 风险说明 |
|---|---|---|---|---|---|

### D. 验收标准映射
| 验收标准 | 对应页面/流程 | 关键操作 | 关键断言 | 备注 |
|---|---|---|---|---|

### E. 自动化风险清单
| 风险点 | 影响 | 触发条件 | 缓解建议 |
|---|---|---|---|

### F. 需我确认的问题（若有）
- 用简洁问题列表输出，不超过 5 条。

## 第二阶段：生成 E2E 用例
当且仅当我回复“继续生成脚本”时，再执行本阶段：

1. 根据第一阶段侦查结果，生成 Playwright 测试代码（TypeScript）。
2. 用例命名必须映射验收标准条目，例如 `AC1`、`AC2`；若验收标准未编号，请先按顺序编号后再映射。
3. 生成内容必须包含：
   - 可复用的前置步骤，例如登录、初始化测试数据、进入目标页面
   - 覆盖验收标准的核心路径
   - 必要的正向断言，以及有价值的负向断言或边界断言
   - 稳定等待策略，禁止使用无意义的硬编码 sleep
   - 与页面结构匹配的组织方式，例如 helper、fixture、Page Object 或直接脚本
4. 输出内容必须包括：
   - 测试文件完整代码
   - 关键设计说明
   - 如有必要，补充测试数据、环境配置或可测试性改进建议
5. 若当前信息不足以安全实现，先列出缺失信息，不得臆造业务规则。
